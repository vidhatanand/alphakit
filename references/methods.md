# Alphakit Image Output Methods

## Codex Image Generation

For prompt-to-image requests, Alphakit uses Codex image generation as the source-image generator, then applies solid-background pair extraction for transparency and deterministic scripts for conversion and verification.

Keep these artifacts separate:

- black-background source generated by Codex image generation
- white-background source generated with the black source as reference input and only the background changed
- green-background source generated only after black/white `negative_diff_ratio` fails threshold
- processed transparent PNG/WebP
- verification report or terminal output

Do not replace generated-image demos with procedural stand-ins. If Codex image generation output is weak, regenerate with a stricter prompt or report the weakness.

## Format Selection

- PNG: Use for transparency, lossless graphics, UI assets, and maximum compatibility.
- JPEG: Use for opaque photographic or illustrative images when smaller file size is desired. JPEG cannot store transparency.
- WebP: Use for web delivery. Use lossless WebP when alpha must be preserved; use quality-controlled lossy WebP for opaque standard images.

Default to true transparent PNG when the user does not specify PNG, JPEG/JPG, WebP, or opacity. If the user explicitly requests a standard or opaque image without naming a format, export an opaque PNG.

# Transparent Image Methods

## Best Method Ranking

1. Aligned black/white background extraction.
2. Green third-background candidate extraction, only after black/white `negative_diff_ratio` fails threshold.
3. Opt-in `soft-photoreal` quality profile, only when the user accepts visual QA and artifact risk for hair, smoke, shadows, glass, or other soft edges.

## Black/White Generation Workflow

Generate the first image on pure black. Read the black file's actual pixel dimensions. Generate the second image with the black image as reference input through Responses image_generation action `generate`, request the actual black-image size, and change only the background to pure white.

Do not create the white image with a new independent text-only prompt. Independent generations usually alter pose, scale, edge detail, lighting, or object placement, which makes extraction invalid.

If reference generation changes anything except the background color or returns a different image size, reject the pair and regenerate. Do not use image edit, chroma-key removal, or procedural art as a fallback.

## Green Third-Background Fallback

Use green fallback only after black/white extraction fails because `negative_diff_ratio` exceeds the configured threshold. Generate a third image with the black image as reference input, actual black-image dimensions, and only the background changed to pure green `#00ff00`.

Run `scripts/add_demo_pair.py --green green.png`. The validator evaluates:

- black-white
- black-green
- green-white

It selects the passing candidate with the lowest `negative_diff_ratio`. Do not use the green image as standalone chroma-key removal.

## Quality Profiles

`strict` is the default. It keeps `negative_diff_ratio <= 0.03`, tight channel-spread limits, and snaps alpha values at or below 20/255 to fully transparent in the saved output so demo backgrounds are genuinely transparent.

`soft-photoreal` is explicit opt-in. It relaxes spread and negative-diff limits for photoreal subjects with hair, soft shadows, glass, smoke, or similar edge complexity, and snaps alpha values at or below 12/255. Because it can accept visible edge or background artifacts, `scripts/add_demo_pair.py` stops with `requires-visual-qa` until rerun with `--visual-qa-pass`.

Do not use `soft-photoreal` as a silent fallback. If the active profile still fails, report the failed metric and stop.

## Black/White Pair Formula

For the same foreground `F` and alpha `a` composited over black and white:

```text
black_rgb = a * F
white_rgb = a * F + (1 - a)
white_rgb - black_rgb = 1 - a
a = 1 - (white_rgb - black_rgb)
F = black_rgb / a
```

This works only when the black and white images are aligned and the foreground is identical.

## Failure Modes

- Two independent generations usually change pose, edge detail, shadows, or object placement.
- Compression, bloom, shadows, and non-solid backgrounds corrupt the formula.
- Edits that change subject pixels, crop, or lighting corrupt the formula.
- Green fallback cannot fix geometry drift; it only offers additional background-pair channels when black/white has excessive negative differences.
- `soft-photoreal` can preserve difficult soft edges but may also preserve speckle, haloing, or subject drift; visual QA is required before committing examples.
- WebP can preserve alpha, but lossy WebP can damage edges. Prefer lossless WebP when WebP is required.
